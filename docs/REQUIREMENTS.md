# Growl 要件定義

# サービス目的

Growlは利用者にまだ見ぬ食体験の提供を目的としています。人々は日々、友人や家族との旅先でグルメ写真を思い出の記録として写真フォルダに記録します。この記録された写真を通じてGrowlはそのユーザーが好む潜在的な食に対する好みを理解します。そしてユーザーがまだ経験したことのない食体験の機会を提供することがこのサービスの目的です。

# 業務要件

## 何を解決するのか

外食や料理をする際、「何を食べたいか・何を作りたいか」という選択は、「検索して探す」か「すでに知っている」かのいずれかに依存しています。しかしユーザー自身が「本当は何を食べたいのか」を言語化することは容易ではなく、潜在的な欲求に辿り着けないまま食体験が終わってしまうことがあります。本サービスは、こうした食に対する漠然とした不満を解消することを目的とします。

## ターゲットユーザー

食事写真が一定量蓄積されているユーザーを主な対象とします。ただし写真が不足している場合は、APIから取得した評価の高い飲食店を初期推薦として提供し、写真の蓄積とともに段階的にパーソナライズされた推薦へ移行する。

## ユースケース

- 自分一人の嗜好で食事をするケース: 一人での食事・旅行先での食探し
- 複数人の嗜好を考慮した食事をするケース: 友人・家族との食事・デート、会食など。相手がGrowlをインストールしている場合は全員の嗜好プロファイルを考慮した推薦が可能。インストールしていない場合は自分の嗜好プロファイルのみで推薦する。

# 機能要件

ここでは代表的な主機能を列挙します。

### 写真解析・嗜好プロファイル生成

カメラロールから食事写真を自動識別し、雰囲気・ジャンル・価格帯域などの属性を抽出して個人の嗜好プロファイルを生成・更新する機能です。

### 飲食店推薦

嗜好プロファイルに基づき飲食店をリストアップする機能。写真が不足している初期状態ではAPIの高評価店をフォールバックとして表示します。

### 料理レシピ推薦

自分で料理を作るのが好きな人向けの機能。嗜好プロファイルに基づき最適な料理レシピを提案します。

### 食体験投稿

料理を撮影・記録し、Growlのプラットフォーム上で他のユーザーと共有することが可能な機能。外食での料理ならば、その料理と店舗名を紐付け推薦機能における飲食店のリストアップ機能の精度向上に役立てる。

# 非機能要件

## 1. 性能・拡張性

### レスポンス時間

- 飲食店・レシピの推薦一覧表示は、リクエストから2秒以内にレスポンスすること。
- 大量のカメラロール解析はバックグランドで行い、UIをブロックしないこと。

### 同時接続数

ここは後で決める

## 2. 可用性・信頼性

個人開発で24時間365日の完全可能は難しいため、許容範囲を決める。

### 稼働時間

- サービス稼働率99%を目標とする。

### データバックアップ

- ユーザーの嗜好プロファイルや投稿データは、RDSの自動バックアップ機能等を用いて1日1回以上バックを取ること。

### フォールバック

- 写真解析APIや外部店名検索APIがダウンしている場合、エラー落ちさせるのではなく、キャッシュデータやデフォルトの「高評価店リスト」を表示すること。

## 3. セキュリティ

ユーザーのプライバシー保護と、信頼性の高いサービス継続を目的とする。

| **項目** | **要件詳細** |
| --- | --- |
| **データ秘匿性** | 通信はすべてTLS 1.2以上（HTTPS）により暗号化すること。 |
| **写真データの保護** | 解析のためにサーバーへアップロードされた画像は、特徴量（ベクトル）抽出完了後、直ちに揮発性メモリまたは一時領域から削除し、永続ストレージに保存しないこと。 |
| **認証・認可** | Firebase Auth等の外部認証基盤を利用し、本人以外の嗜好プロファイルや非公開投稿へのアクセスを厳格に制限すること。 |
| **APIキー管理** | Google Places APIやOpenAI API等の各種キーはソースコードに含めず、AWS Secrets Managerや環境変数を用いて動的に管理すること。 |
| **法的遵守** | 日本国内の個人情報保護法およびGDPR等の基準に準じ、写真解析に関する同意取得フローを実装すること。 |

## 4. 運用・保守性要件

開発者一人のリソースで、持続的にサービスを改善・維持できる体制を構築する。

| **項目** | **要件詳細** |
| --- | --- |
| **ログ管理** | アプリケーションログ（FastAPI等）およびアクセスログを中央管理し、エラー発生時のスタックトレースを後日確認可能にすること。 |
| **死活監視・通知** | サーバーダウンやAPIエラーの急増を検知した場合、即座に開発者（Slack/Discord等）へ通知が飛ぶ仕組みを構築すること。 |
| **コスト監視** | AWSの月次予算アラートを設定し、想定以上のトラフィックやAPI利用が発生した場合に早期発見できるようにすること。 |
| **CI/CD（自動化）** | GitHub Actions等を利用し、コードのプッシュからテスト・デプロイまでを自動化することで、手動操作による事故を防止すること。 |
| **DBメンテナンス** | Amazon RDSのマネージド機能を活用し、日次の自動バックアップおよびマイナーバージョンの自動パッチ適用を有効にすること。 |
| **データ鮮度管理** | 推薦機能で利用する店舗情報のリンク切れや閉店情報を、ユーザー報告に基づき簡易的にフラグ立て（論理削除）できる管理機能を備えること。 |
