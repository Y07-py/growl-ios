# Growlのドメイン

## 主要ドメインについて
Growlがビジネス的な視点で見た時に、どのような問題領域を解決しようとするのかを定義します。

### 1. コアドメイン（Core Domain）
Growlの競争優位性と独自性を生み出す、最も重要な問題領域です。

| ドメイン名 | 責務（何を解決するか） | 主要概念（扱うデータ・ルール） |
| :--- | :--- | :--- |
| **嗜好解析** | 飲食関連の写真を解析し、ユーザーの「好き」を構造化データへ変換する。 | 嗜好プロファイル、画像特徴量、食の属性（雰囲気・ジャンル・価格帯） |
| **推薦エンジン** | 解析プロファイルと外部データを照合し、最適な食体験をマッチングする。 | 推薦スコアリング、フォールバック戦略、グループ推薦ロジック |

### 2. サブドメイン（Sub-Domain）
コアドメインを支える、あるいは一般的なサービス運営に必要となる問題領域です。

| ドメイン名 | 責務（何を解決するか） | 主要概念（扱うデータ・ルール） |
| :--- | :--- | :--- |
| **コンテンツ** | 外部APIの店舗情報や、自前のレシピデータの管理・提供を行う。 | 飲食店情報、メニュー、レシピ、営業ステータス |
| **ソーシャル** | 食体験の投稿・共有、および他ユーザーとの繋がりを管理する。 | 投稿、共有設定、嗜好プロファイルの共有 |
| **アイデンティティ** | ユーザー認証・認可、およびサブスクリプション・決済を管理する。 | ユーザーアカウント、権限管理、決済管理 |

## エンティティ
上記で定義されたドメインをソフトウェアから見て利用できる概念にカプセル化します。

### 1. 嗜好解析・ドメイン

| エンティティ | 責務 | 主要な属性 | 
| :--- | :--- | :--- |
| **TasteProfile** | ユーザーの食の好みを表現する中心的なエンティティ。| ユーザーID、属性ベクトル、更新日、解析済み件数 | 
| **AnalyzedImage** | 解析が完了した1枚の画像データ | 画像メタデータID、抽出されたタグ、信頼度スコア |
| **TasteFeature** | 嗜好を構成する個別の特徴。「辛いもの好き」などの嗜好を表現する最小単位 | 特徴カテゴリ、重み、有効期限 | 

### 2. 推薦エンジン・ドメイン

| エンティティ | 責務 | 主要な属性 | 
| :--- | :--- | :--- |
| **Recommendation** | ユーザーに提示される「推薦結果」そのもの。 | 推薦対象ID、マッチングスコア | 
| **RecommendationCriteria** | 推薦を行う際の条件。| 参加人数、予算範囲、利用シーン、現在の位置情報 | 
| **GroupProfile** | 複数人のTasteProfileを合成した一時的なプロファイル | メンバーIDリスト、合成済み属性ベクトル |

### 3. コンテンツ・ドメイン

| エンティティ | 責務 | 主要な属性 | 
| :---- | :--- | :--- |
| **Restaurant** | 飲食店の情報を定義する。 | 店舗名、位置情報、ジャンル、価格帯、営業時間、ステータス | 
| **Recipe** | 料理レシピの情報を定義する。| 料理名、材料リスト、手順、所用時間、難易度 | 

### 4. ソーシャル・ドメイン
5. 
| エンティティ | 責務 | 主要な属性 | 
| :--- | :--- | :--- | 
| **Post** | ユーザーが投稿する食体験の記録 | 投稿者ID、関連店舗ID、写真URL、公開範囲設定 | 
| **SharedConnection** | ユーザー間の繋がりや嗜好共有の許可状態。 | ユーザーA/B、共有権限、承認ステータス | 

### 5. アイデンティティ・ドメイン
| エンティティ | 責務 | 主要な属性 | 
| :--- | :--- | :--- | 
| **Account** | ユーザーの基本情報と契約状態 | ユーザーID、メールアドレス、プラン状態、登録日 |
